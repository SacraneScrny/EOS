# PerformanceReadme — EOS Core

Документ о производительности ядра EOS: на чём оно построено, где «горячо», чего ожидать,
с чем сравнивать и как писать код, который не упирается в рантайм. Всё ниже выведено из
самого кода (`Storage<T>`, `SystemsRunner`, `World`, `EntityQuery`, `EventChannel<T>`,
`TagsContainer`, `HierarchyContainer` и т.д.), а не из общих соображений.

> Парный документ для Unity-моста — `EOS.Unity/PerformanceReadme.md`.

---

## 1. Философия и цели производительности

EOS — **однопоточный** ECS, спроектированный под линейный, cache-friendly доступ к данным:

- **Компоненты лежат в плотных массивах** (`Storage<T>` — dense-array sparse-set), а не в
  словарях. Итерация системы — это проход по непрерывному `T[]`, а не по хеш-таблице.
- **Ноль зависимостей, ноль ссылок на движок.** Ядро — чистая C#-библиотека. Это значит
  предсказуемую стоимость операций и отсутствие интеропа на горячем пути.
- **Аллокации вынесены на этап конфигурации, а не на кадр.** Структурные перечислители,
  struct-события, struct-сущности; рост массивов — амортизированное удвоение.
- **Две взаимозаменяемые модели исполнения** — рефлексия (zero-config) и кодоген
  (zero-alloc). Это главный рычаг производительности (раздел 5).

Чего EOS **не** делает: не распараллеливает системы, не использует SIMD/Burst, не делает
архетипную пересборку (как Unity DOTS). Это сознательный размен: простота и
детерминизм против пикового throughput на миллионах сущностей.

## 2. Модель памяти и аллокаций

### 2.1 `Storage<T>` — плотный sparse-set

Каждый тип компонента хранится в `Storage<T>` как набор **параллельных массивов**,
индексируемых плотным слотом:

```
_data[]          T[]        — сами компоненты (непрерывно, 0..Count)
_owners[]        int[]      — id владельца слота
_ownerVersions[] ushort[]   — версия владельца (детект устаревших хэндлов)
_addVersion[]    ulong[]    — watermark канала [New]
_markVersion[]   ulong[]    — watermark канала [Bumped]
_enableVersion[] / _disableVersion[]  — каналы [Enabled]/[Disabled]
_ready[]         bool[]     — готов ли (Awake+Start+enabled+active)
_sparse[]        int[]      — entity.Id → плотный слот
```

- **Add** — O(1): берёт хвостовой слот, заполняет, обновляет `_sparse`.
- **Lookup (`IndexOf`)** — O(1): `_sparse[id]` + валидация `_owners`/`_ownerVersions`.
- **Remove** — O(1) **swap-remove**: удаляемый слот затирается последним, `_sparse`
  чинится для перемещённого хвоста. Массив остаётся непрерывным → итерация быстрая.
- **Рост** — удвоение от `InitialCapacity = 16`. Все параллельные массивы растут вместе
  одним `Array.Resize`. Амортизированно O(1), но см. практику прогрева (раздел 11).

Следствие для итерации: проход системы по `Storage<T>.All` — это линейный проход по
`T[]` без косвенности и без cache-miss на каждом элементе (только сам `T` — ссылочный
объект, см. ниже).

### 2.2 Что всё-таки ссылочное

`EosObject` — **класс**, не struct. То есть `_data[i]` — это ссылка, и обращение к полям
компонента — это разыменование в кучу. Плотный массив даёт быструю последовательность
**слотов** и хорошую локальность заголовков, но поля компонента живут там, где их
положил GC. Для горячих числовых данных это нюанс: храните «горячие» поля компактно в
самом компоненте, не размазывайте состояние по вложенным объектам.

### 2.3 Где аллокации происходят (и где нет)

| Операция | Аллокации |
|---|---|
| Итерация системы (кодоген) | **ноль** |
| Итерация системы (рефлексия) | `object[]` аргументов + бокс на каждый матч |
| `world.Query<T>()` + `foreach` | **ноль** (struct-перечислитель) |
| `world.Query<T>().With<U>()` | малый alloc массива фильтра **на конфигурации**, не на проходе |
| `World.Event(in T)` | копия struct в буфер, без бокса (бокс только в рефлексия-консьюмере) |
| Создание/удаление сущности | амортизированно ноль (переиспользование id + стек free) |
| Теги `Has*`/`Match*` | ноль (битовые маски, scratch-список переиспользуется) |
| `Storage` рост массива | разовый `Array.Resize` при превышении ёмкости |

## 3. Кадровый цикл — что выполняется и в каком порядке

`World.Update(dt)` (см. `World.cs`):

```
NextFrame
→ BeforeAll.Execute → BeforeUpdate.Execute     (ECB — отложенные структурные изменения)
→ Events.Promote + Events.Trim                 (staging→live, ретайр прочитанного/устаревшего)
→ InitializeSystems.Run                        (Awake+Start ожидающих, затем MarkReady)
┌─ iteration guard ─────────────────────────
│ → Systems.UpdateEvents                       (EventExecute)
│ → Systems.Update                             (Execute)
│ → Objects.Update                             (IObjectUpdate)
└───────────────────────────────────────────
→ AfterUpdate.Execute
```

`FixedUpdate`/`LateUpdate` — та же форма, но **без** `InitializeSystems` и без
`BeforeAll`/`AfterAll`. Практический вывод: если гнать мир только через `FixedUpdate`,
новые компоненты никогда не инициализируются — это не баг, а форма цикла.

**Стоимость самого каркаса кадра близка к нулю**: пустой мир — это несколько проверок
флагов, инкремент кадра и пустые проходы по спискам систем. Вся цена — в телах систем,
в `InitializeSystems` (когда есть что инициализировать) и в выполнении ECB.

### Авто-инструментирование профайлером

Каждая фаза и **каждое тело системы** обёрнуты в `EosProfiler.Sample(label)`. По
умолчанию профайлер выключен (`Enabled = false`, `NullProfilerBackend`) — это
`readonly struct Scope` с пустыми `Begin/End`, накладные расходы пренебрежимы. Включать
бэкенд только при замерах (раздел 12).

## 4. Горячий путь №1 — исполнение систем

Тело системы строится один раз при `World.Init()` из формы её `Execute(...)`. На кадре
выполняется уже готовый делегат `Action<float, ulong>`.

### Что делает построенный запрос (concrete-путь, `BuildConcreteQuery`)

1. **Выбор пивота** — среди обязательных concrete-параметров берётся storage с
   **наименьшим `Count`**. Итерируем по нему, остальные компоненты проверяем точечным
   `IndexOf`. Это ключевая оптимизация многокомпонентных запросов: цена ≈ размер
   наименьшего множества, а не произведение.
2. **Ready-гейт** — `IsReady(i)` отсекает не-готовые/выключенные слоты сразу.
3. **Фильтры** — `[Include]`/`[Exclude]` (has/not-has) и теги; проверяются до вызова тела.
4. **Interface fan-out** — для интерфейсных параметров строится декартово произведение
   реализаций (`[Each]`), затем тело вызывается для каждой комбинации.

### Рефлексия против кодогена — цена матча

Это самое важное место документа.

- **Рефлексия (по умолчанию).** На **каждый матч** строится `object[] args`
  (`BuildArgs`), а сами компоненты достаются через `MethodInfo.Invoke` с ещё одним
  `new object[]{ entity }`. Каждый фильтр `HasReady` — тоже `MethodInfo.Invoke`. Это
  корректно и не требует настройки, но даёт **бокс + несколько аллокаций и
  reflection-invoke на каждую обработанную сущность**.
- **Кодоген (`SystemRegistryGenerator`, opt-in).** Генерируется типизированное тело:
  прямые вызовы `Get<Concrete>()`, прямой `self.Execute(c0, c1, …)`, без `object[]`, без
  бокса, без `MethodInfo.Invoke`. **Ноль аллокаций на проходе.**

Ориентир: для системы, обрабатывающей тысячи сущностей каждый кадр, разница между двумя
путями — это разница между «заметный GC-трафик и reflection-оверхед» и «чистый линейный
проход». Формы, которые кодоген типизировать **не** может (`SystemShape.CanTypeBody`
возвращает false — например, `[Budget]`, или интерфейсный `[Removed]`), автоматически
падают в рефлексию с предупреждением; всё работает, просто без оптимизации.

**Практика:** прототипируйте на рефлексии, перегенерируйте реестр (`Create System
Registry` в Unity-мосте) перед профилированием и релизом. Реестр сам себя ставит через
`[ModuleInitializer]`, переключая `SystemsRunner` на типизированные тела.

### Гейтинг систем — дешёвые способы «не выполнять»

- `UpdateWhen()` / `IsEnabled` / `On()`/`Off()` — рантайм-гейт всей системы.
- `[Group(...)]` + `SystemGroups.Disable<T>()` — иерархическое отключение пачки систем
  одной проверкой `IsEnabled(group)` (она вшита в `isUpdate` делегат).
- **Важно про реактивные:** курсор реактивной системы продвигается **каждый кадр даже у
  выключенной группы** — события за период выключения теряются (дропаются). Если они
  должны накапливаться — гейтьте через `[Delay]`, а не через выключение группы.

## 5. Горячий путь №2 — реактивные каналы

`[New]`, `[Bumped]`, `[Enabled]`, `[Disabled]`, `[Removed]` — реактивные системы,
которые **не сканируют всё**, а сравнивают версии с курсором.

- **Дешёвый ранний выход.** Перед сканом — `ChannelMax(driver, channel) <= cursor →
  return`. Если в этом канале storage'а с прошлого раза ничего не стампилось, система
  не делает **ничего** (одна проверка `ulong`). Это делает реактивные системы почти
  бесплатными в кадрах без событий.
- **Watermark-арифметика.** Каждый stamp (`MarkReady`/`Bump`/enable/disable/remove)
  двигает `World.Version` и пишет per-slot версию; система проходит storage и берёт
  слоты, чьи версии `> cursor`.
- `[Disabled]` — единственный канал, который сканирует **не-готовые** слоты (по
  построению). Остальные сначала проверяют `IsReady`.
- `[Removed]` читает per-storage **removal log** (кольцо id+version+cascade),
  трим по возрасту кадров (`World.ReactiveRetentionFrames`, дефолт 16). Курсор каждой
  системы гарантирует read-once, остаточные записи безвредны.

**Вывод:** предпочитайте `[New]`/`[Removed]` опросу «появилось ли что-то» в обычной
системе. Реактивная система в спокойном кадре — это сравнение двух чисел.

### `explicit` против `cascade`

`[Enabled]`/`[Disabled]`/`[Removed]` по умолчанию реагируют только на **явные** рёбра
(`Enable/Disable`, `Remove<T>()`). Каскадные рёбра (деактивация родителя, `Destroy`
поддерева) включаются флагом `includeCascade: true`. Не включайте каскад без нужды — это
расширяет множество, по которому система реагирует.

## 6. Горячий путь №3 — императивные запросы `world.Query<...>()`

Для кода **вне** цикла систем (UI, MonoBehaviour, тулзы). Возвращает `readonly struct`
с **alloc-free** struct-перечислителем:

```csharp
foreach (var h in world.Query<Health>()) { ... }                 // ноль аллокаций
foreach (var (p, v) in world.Query<Position, Velocity>()) { ... } // пивот на меньший storage
```

- Тот же ready-гейт, что у систем; дедуп по сущности; пивот на меньший storage для
  мультикомпонентных.
- **Fluent-фильтры (`.With/.Without/.WithTag/...`)** копируют массив фильтра — это малый
  alloc **на конфигурации запроса**, перечисление остаётся alloc-free. Если строите
  запрос с фильтрами в горячем цикле каждый кадр — соберите его один раз и
  переиспользуйте, либо считайте этот alloc.
- Терминалы `Any()/First()/TryFirst()` выходят на первом матче — не материализуют.
  `ToList()` — единственный, кто аллоцирует список под результат (по запросу).
- Перечисление держит iteration-guard, поэтому структурные изменения во время `foreach`
  ловятся `StructuralChangePolicy` так же, как в системах.

## 7. События

`EventChannel<T>` — per-type буфер struct-событий:

- `Event(in T)` копирует struct в staging (без бокса), рост staging — удвоение.
- `Promote` переносит staging→live с возрастающими seq; `Trim` ретайрит по **min-cursor**
  (событие живёт, пока все консьюмеры не прочли) или по возрасту.
- **Read-once** держится курсором на каждый `EventExecute` — мультитик `FixedUpdate` и
  порядок фаз не приводят к двойному чтению.
- Рефлексия-консьюмер **боксит** значение (`BoxedAt`); кодоген берёт `ValueAt` без бокса.
  Ещё одна причина прогенерировать реестр.

События дешевле компонентов для «выстрелил-и-забыл» сигналов: нет сущности, нет
структурного изменения, безопасно из середины итерации.

## 8. Теги — битовые маски

`TagsContainer` — per-entity битовая маска (`ulong[]`, индексируется `id * _words`):

- Ключи (строки/enum) интернируются в индексы бит один раз (`TagRegistry`).
- `BuildMask` компилирует набор тегов в `ulong[]`-маску; `MatchAll/None/Any/One` —
  это цикл по словам с битовыми операциями (`MatchOne` использует SWAR-popcount).
- Системные тег-фильтры компилируют маску **один раз при Init**, на кадре — только
  битовый матч. Очень дёшево.

**Практика:** теги — это правильный инструмент для «грубой» классификации (фракция,
состояние, слой). Они дешевле, чем маркер-компоненты, когда не нужны данные; но
компонент-маркер участвует в пивоте запроса (раздел 4), а тег — нет. Если по признаку
часто **итерируют небольшое подмножество**, маркер-компонент даст лучший пивот; если
признак — лишь фильтр поверх другого запроса, берите тег.

## 9. Иерархия — O(1) чтения

`HierarchyContainer` — sparse-массивы (`_parents`/`_firstChild`/`_nextSibling`/
`_prevSibling` + `_childCount` + `_branchActive` + `_root`):

- `GetParent`/`GetRoot`/`ChildCount`/`Children()` — **O(1)**, без alloc. `GetRoot` —
  прямой `_root[id]`, не обход предков.
- Единственная O(subtree)-операция — **репарент**: `SetParent`/`Detach` проталкивают
  новый корень и эффективный active вниз по перемещаемому поддереву.
- Репарент — **не** структурное изменение (разрешён из середины итерации), эмитит
  `ParentChanged`.
- `Destroy` каскадит поддерево (дети раньше родителя).

**Практика:** глубокие частые репаренты больших поддеревьев — единственное «дорогое»
место иерархии. Эффективный active кэшируется, поэтому `IsActive` остаётся O(1) даже в
глубоком дереве.

## 10. Пулинг компонентов

Opt-in на тип через маркер `IPoolableObject`. `Storage<T>` держит `Stack<T>`: `Remove`
после `Dispose` возвращает инстанс в пул (через `ResetForReuse`), `Add` арендует вместо
`new T()`. Переиспользованный инстанс **проходит полный цикл заново** (`Awake`/`Start` →
`MarkReady`), `[New]` перевыстреливает — пулинг прозрачен.

Когда включать: типы с **высоким churn** (часто добавляются/удаляются — снаряды,
партиклы, эффекты). Без маркера каждый `Add` — это `new T()` и работа для GC. С маркером
— переиспользование без GC-давления.

Цена: сброс полей. `OnRent`/`OnReturn` хуков **нет** (в отличие от Unity-вью); сбрасывайте
устаревшие данные в `OnDispose` или полностью инициализируйтесь в `OnAwake`.

## 11. Сценарии и ожидания

| Сценарий | Что доминирует | Ожидание / рекомендация |
|---|---|---|
| **Много сущностей, простая логика** (10k+ движущихся точек) | проход тел систем | Обязателен **кодоген**. Линейный проход по плотному массиву; держите горячие поля в компоненте. |
| **Мало сущностей, тяжёлая логика** | сами вычисления | Путь исполнения не важен; рефлексия приемлема. |
| **Высокий churn** (спавн/деспавн каждый кадр) | `Add`/`Remove`, `InitializeSystems` | Включите `IPoolableObject`. Спавн откладывайте в ECB; не создавайте/не удаляйте из середины итерации. |
| **Редкие события на больших данных** | реактивный ранний выход | `[New]`/`[Removed]`. В спокойных кадрах — почти бесплатно (сравнение `ulong`). |
| **Тяжёлая периодическая работа** (AI-тик, регенерация) | равномерность нагрузки | `[Delay]`/`[DelayFrame]` для каденции, `[Budget(n)]` для тайм-слайсинга (квота в кадр, остальное переносится). |
| **Многокомпонентный запрос** | размер пивота | Делайте обязательным **самый редкий** компонент — пивот выберет его, цена ≈ его `Count`. |
| **Старт/загрузка большого мира** | рост массивов, инициализация | Прогрейте storage заранее (создайте N сущностей до геймплея), чтобы амортизировать `Array.Resize`. |
| **Холодный `World.Init()`** | discovery + сортировка | Рефлексия сканирует все сборки; кодоген-провайдер отдаёт готовый список (быстрее и без скана). Разовая цена, не на кадр. |

### Про `[Budget]` и путь исполнения

`[Budget]` **всегда** идёт по рефлексии (`CanTypeBody` → false), потому что это
throttle-тяжёлая логика, где бокс пренебрежим относительно сэкономленной работы.
Не-реактивные budget-запросы round-robin'ят персистентный offset (честно,
eventually-complete). Реактивный budget с **одним** реактивным параметром обрабатывает
рёбра по возрастанию версии и двигает курсор только за обработанным. Несколько
реактивных параметров + budget → предупреждение и запуск **без** бюджета (чтобы не
потерять ребро).

## 12. Профилирование

- `EosProfiler` — статический фасад над `IEosProfilerBackend`. Выключен по умолчанию.
- Включите `AggregatedProfilerBackend` для замеров: он копит ticks/calls по меткам;
  `Dump(reset)` отдаёт отчёт, отсортированный по суммарному времени.
- Метки уже расставлены: `World.Update`/`FixedUpdate`/`LateUpdate`, `InitializeSystems`,
  `Systems.Update`, `Objects.Update`, и **каждое тело системы** (метка = имя типа).
- Unity-мост подключает `UnityProfilerBackend` (`Begin → ProfilerMarker`) — метки видны в
  Unity Profiler.

Что смотреть: метки систем с наибольшим суммарным временем; всплески в
`InitializeSystems` (значит, большой батч спавнов в один кадр — разнесите через ECB/budget);
GC-аллокации на кадре (значит, горячая система всё ещё на рефлексии).

## 13. Сравнения

| | EOS | Традиционный GameObject/`Update()` | Архетипный ECS (Unity DOTS) |
|---|---|---|---|
| Хранение | плотные параллельные массивы на тип | объекты по куче | архетипные чанки (SoA) |
| Итерация | линейный проход по слоту, дедуп по пивоту | виртуальные вызовы вразброс | линейно по чанку, SIMD/Burst |
| Аллокации/кадр | ноль (кодоген) / бокс (рефлексия) | зависит | ноль |
| Многопоточность | нет (однопоточный) | нет | да (jobs) |
| Структурные изменения | через ECB, swap-remove O(1) | мгновенно, дорого | через ECB, пересборка архетипа |
| Порог входа | низкий (POCO-компоненты, рефлексия из коробки) | низкий | высокий |
| Where it wins | детерминизм, простота, отсутствие интеропа, «тысячи-десятки тысяч» сущностей | сцены с малым числом активных объектов | «сотни тысяч-миллионы» с тяжёлой числовой работой |

EOS целит в нишу между «всё на MonoBehaviour» и «полный DOTS»: вы получаете
data-oriented раскладку, реактивность и zero-alloc исполнение без перехода на Burst,
структуры-only и job-граф.

## 14. Хорошие практики (чек-лист)

**Раскладка данных**
- Горячие числовые поля держите прямо в компоненте, не за вложенными ссылочными объектами.
- Маркер vs тег: если по признаку часто **итерируете подмножество** — маркер-компонент
  (хороший пивот); если признак — лишь фильтр — тег (дешевле).

**Запросы**
- Обязательным делайте **самый редкий** компонент → лучший пивот.
- В мультикомпонентных запросах не добавляйте необязательные/интерфейсные параметры, если
  их можно выразить через `[Include]`/тег-фильтр.
- Императивные запросы с fluent-фильтрами не пересоздавайте в горячем цикле — стройте раз.

**Исполнение**
- Перегенерируйте реестр систем (кодоген) перед профилированием/релизом. Рефлексия — для
  разработки.
- Тяжёлую периодику — на `[Delay]`/`[DelayFrame]`; «размазать пик» — на `[Budget]`.
- Реактивные `[New]`/`[Removed]` вместо опроса «появилось/исчезло».

**Структурные изменения**
- Из систем — только через ECB (`AfterUpdate`/`AfterCurrentPhase`). Прямое изменение под
  guard'ом бросает по политике `Throw`.
- Высокий churn типа → `IPoolableObject` + сброс полей в `OnDispose`.
- Спавн большого батча разносите по кадрам (иначе `InitializeSystems` даст пик).

**Жизненный цикл**
- Прогревайте storage/сущности на загрузке, чтобы амортизировать рост массивов.
- Не гоняйте мир только через `FixedUpdate`, если ждёте инициализации компонентов
  (она только в `Update`).

## 15. Антипаттерны (что бьёт по производительности)

- **Релиз на рефлексии** для систем с тысячами матчей — лишний бокс и reflection-invoke
  на каждую сущность каждый кадр.
- **Создание/удаление компонентов из середины итерации** напрямую — нарушает guard;
  правильный путь — ECB, swap-remove делает удаление дешёвым только вне прохода.
- **`world.Query<>().With<>()...` каждый кадр в цикле** — копирование массивов фильтра на
  каждой конфигурации.
- **Спавн всего мира в один кадр** — пик в `InitializeSystems`; разносите.
- **Каскадные реактивные каналы без нужды** (`includeCascade: true`) — расширяют
  множество срабатываний.
- **Глубокие частые репаренты больших поддеревьев** — единственная O(subtree)-операция
  иерархии.
- **Состояние компонента, размазанное по вложенным объектам** — теряется локальность,
  которую даёт плотный массив.

---

### TL;DR

Плотные массивы + sparse-set дают O(1) на всё и линейную итерацию. Главный рычаг —
**кодоген вместо рефлексии** (zero-alloc на проходе). Реактивные каналы почти бесплатны в
спокойных кадрах. Структурные изменения — через ECB. Пулинг — на типы с высоким churn.
Профилируйте через `EosProfiler`/Unity Profiler и смотрите на метки систем и GC на кадре.
