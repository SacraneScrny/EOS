## 1. Что это вообще такое

EOS — это **ECS-фреймворк**. ECS расшифровывается как Entity-Component-System
(сущность–компонент–система). Это способ организовать игровой код, при котором:

- **Сущность (entity)** — просто номер, «бирка». Сама по себе она ничего не умеет
  и ничего не хранит. Игрок, пуля, дверь — всё это сущности.
- **Компонент (component)** — кусочек данных, прицепленный к сущности. Например,
  компонент `Health` с полем `Value`, компонент `Position` с координатами.
  У компонента нет «поведения кадра за кадром» в классическом смысле — это в первую
  очередь данные.
- **Система (system)** — кусок логики, который каждый кадр пробегает по всем сущностям
  с нужным набором компонентов и что-то с ними делает. Например, «для каждой сущности,
  у которой есть `Position` и `Velocity`, сдвинь позицию».

Чем это отличается от привычного Unity-подхода «повесим MonoBehaviour на GameObject»?
Тем, что данные и логика разведены по разным местам, а данные одного типа лежат
в памяти **плотным массивом, подряд**. Когда система пробегает по тысяче компонентов
`Position`, процессор читает память последовательно — это быстро. В описании ядра это
называется «designed for cache-friendly access patterns»: компоненты в плотных
массивах, обход линейный.

Вторая ключевая идея EOS: **ядро не знает про Unity вообще**. В этом репозитории нет
ни одного упоминания `UnityEngine`. Нет даже `.csproj` — это просто папка с C#-файлами,
которую кладут в Unity-проект рядом со «слоем-мостом» EOS.Unity. Ядро отвечает за
данные и логику, а всё «зримое» (префабы, GameObject-ы, PlayerLoop) живёт в EOS.Unity
и подключается к ядру через специально оставленные «швы» — публичные точки
подключения. Эти швы мы встретим по ходу рассказа:

| Шов | Что подключает |
|---|---|
| `Universe.Update/FixedUpdate/LateUpdate` | кто-то снаружи должен дёргать ядро каждый кадр |
| `IncarnationBridge` | как по идентификатору создать визуальный объект |
| `WorldBootstrap.Provider` | как «засеять» каждый новый мир стартовыми данными |
| `EosLog.OnLog` | куда писать логи |
| `EosProfiler.Backend` | куда слать замеры производительности |
| `WorldLoader.OnSave/OnLoad` | куда сохранять и откуда грузить снимки мира |

Ещё одна важная вещь про стиль: в ядре **нет комментариев вообще** — это осознанное
правило репозитория. Весь «рассказ» о том, как оно работает, — это данный документ и
файл `CLAUDE.md`. Ошибки никогда не глотаются молча: везде `try/catch` плюс запись
в `EosLog`. Недопустимые операции (обновление без запуска, регистрация сервиса посреди
кадра) пишут ошибку в лог и прекращаются, не роняя игру. Единственное исключение —
защита от структурных изменений посреди итерации, она по умолчанию бросает исключение
(подробнее в главе 14).

И последнее: ядро **однопоточное**. Никаких джобов и параллельности — всё происходит
последовательно, в одном потоке, что сильно упрощает рассуждения о порядке событий.

## 2. Карта репозитория

```
EOS/
├── Core/            Universe (корень всего), World (один мир), контекст, сервисы,
│                    политика структурных изменений
├── Entities/        EosEntity (идентификатор сущности) и контейнер живых сущностей

├── Objects/         EosObject — базовый класс компонентов, Incarnation (шов на вью)
├── Storage/         Storage<T> — плотное хранилище компонентов одного типа
├── Systems/         EosSystem, исполнитель систем, группы, командные буферы
├── Queries/         World.Query<…> — запросы к миру снаружи систем
├── Events/          одно-кадровые события-структуры
├── Tags/            теги — битовые метки на сущностях
├── Attributes/      атрибуты для систем: [Group], [New], [WithTag] и т.п.
├── Extensions/      методы-расширения: entity.Add<T>(), entity.SetParent() и т.п.
├── Serialization/   снимки мира, сохранение/загрузка
├── CodeGen/         генератор быстрого кода для систем
├── Loader/          швы для внешнего слоя: WorldBootstrap, IncarnationBridge,
│                    EosDomainReset
├── Logging/         EosLog
├── Profiling/       EosProfiler
└── Diagnostics/     текстовые дампы мира для отладки
```

## 3. Вселенная и миры

Файлы: `Core/Universe.cs`, `Core/World.cs`, `Core/WorldBound.cs`.

### Universe — статический корень

`Universe` — статический класс, единая точка входа. Он владеет одним **миром по
умолчанию** (default world) и списком дополнительных миров. Жизненный цикл такой:

```csharp
Universe.Boot();          // один раз при старте игры

// каждый кадр (в Unity это делает EOS.Unity автоматически):
Universe.Update(deltaTime);
Universe.FixedUpdate(fixedDeltaTime);
Universe.LateUpdate(deltaTime);

Universe.Shutdown();      // при выходе
```

`Boot()` (см. `Core/Universe.cs:30`) сначала уничтожает предыдущие миры, если они
были — то есть повторный вызов безопасен, — создаёт свежий мир по умолчанию и
инициализирует его. Затем происходит интересная вещь: **автозагрузка сохранения**.
`Boot()` вызывает внешний хук `WorldLoader.OnLoad` и, если тот вернул снимок мира,
восстанавливает его через `WorldSerializer.Restore`. То есть «загрузка сейва при
старте» встроена прямо в запуск — внешнему слою достаточно подвесить функцию на
`WorldLoader.OnLoad`.

Флаги состояния:

- `IsBooted` — был ли вызван `Boot()`. Попытка обновляться без запуска — ошибка в лог.
- `IsEnabled` — включена ли вселенная. `Off()` ставит игру «на паузу» (обновления
  молча пропускаются), `On()` возобновляет.
- `IsIterating` — идёт ли прямо сейчас обход (мы внутри `Update`/`FixedUpdate`/
  `LateUpdate`). Пока он истинен, **управление мирами запрещено**: нельзя создавать,
  уничтожать, сбрасывать миры и переключать `On/Off` — будет ошибка в лог. Это защита
  от того, чтобы мир исчез у самого себя из-под ног посреди кадра.

### Дополнительные миры

Обычно достаточно одного мира, но можно завести ещё — например, отдельный мир для
меню или для фоновой симуляции:

```csharp
var menuWorld = Universe.CreateWorld("menu", isSerializable: false);
Universe.TryGetWorld("menu", out var w);
Universe.DestroyWorld(menuWorld);
```

Ключи миров уникальны, мир по умолчанию уничтожить нельзя. Мир с
`IsManualUpdate = true` вселенная пропускает — его нужно обновлять самому, вызывая
`world.Update(dt)` напрямую.

### Tick — обновление с фиксированным шагом

`Universe.Tick(realDelta, fixedStep = 1/60, maxSteps = 8)` — альтернативный способ
крутить вселенную, когда нужна симуляция с постоянным шагом (например, для
детерминизма). Внутри — накопитель: реальное прошедшее время складывается, и пока
накопилось больше одного шага, вызывается `Update(fixedStep)` (заметьте: именно
`Update`, а **не** `FixedUpdate` — фаза `FixedUpdate` срабатывает только если вы
зовёте её сами). Не больше `maxSteps` шагов за раз; если игра «провисла» так, что
долг всё равно больше шага, остаток просто отбрасывается — лучше потерять время,
чем уйти в спираль смерти, когда каждый кадр должен догонять предыдущий.

### World — один мир

`World` (`Core/World.cs`) — это «коробка», в которой живёт вся подсистемная начинка:

```csharp
public EntitiesContainer Entities { get; }        // живые сущности
public ObjectsContainer Objects { get; }          // реестр компонентов
public SystemsRunner Systems { get; }             // исполнитель систем
public TagsContainer Tags { get; }                // теги
public HierarchyContainer Hierarchy { get; }      // родитель-ребёнок
public ObjectsStorageMap ObjectsStorages { get; } // хранилища компонентов по типам
public SystemGroups SystemGroups { get; }         // группы систем
public EventsContainer Events { get; }            // события
```

Каждая из этих подсистем наследует `WorldBound` (`Core/WorldBound.cs`) — простую
базу, которая получает ссылку на свой мир через `Init(World)`. Так подсистемы
дотягиваются друг до друга через общий мир, а не через статические поля.

Наружу `Universe.DefaultWorld` отдаётся как `IReadOnlyWorld` — интерфейс «только
чтение». Полноценный `World` для мира по умолчанию — внутренний
(`Universe.InternalDefaultWorld`). Это намеренное ограничение: случайный код не
должен мочь, скажем, вызвать `Dispose` у мира по умолчанию. (В Unity-слое это
ограничение обходится легально: ядро и мост компилируются в одну сборку, поэтому
`internal` оттуда виден.)

## 4. Кадр изнутри: что происходит за один Update

Файл: `Core/World.cs:220` (метод `Update`), а также `FixedUpdate` и `LateUpdate` ниже.

Это самая важная глава для понимания «почему оно сработало не в том порядке».
Вот что происходит при вызове `World.Update(dt)`, шаг за шагом:

```
CurrentPhase = Update; номер кадра +1
→ BeforeAll.Execute        — выполнить отложенные команды (только в Update!)
→ BeforeUpdate.Execute     — ещё один буфер отложенных команд
→ Events.Promote + Trim    — «свежие» события становятся видимыми, старые удаляются
→ InitializeSystems.Run    — у новых компонентов вызываются Awake и Start
┌─ начало «охраняемой зоны» (iteration guard) ─────────────
│ → Systems.UpdateEvents   — методы EventExecute всех систем
│ → Systems.Update         — методы Execute всех систем
│ → Objects.Update         — компоненты с интерфейсом IObjectUpdate
└─ конец охраняемой зоны ──────────────────────────────────
→ AfterUpdate.Execute      — отложенные команды «после кадра»
```

Разберём непонятные слова.

**Командные буферы (`BeforeAll`, `BeforeUpdate`, `AfterUpdate`…)** — очереди
отложенных операций «создать сущность», «добавить компонент», «удалить». Их подробно
разберём в главе 14. Здесь важно одно: они выполняются в строго определённых точках
кадра, **вне** охраняемой зоны.

**Охраняемая зона (iteration guard)** — пока выполняются системы, мир считается
«в процессе обхода» (`World.IsIterating == true`). В это время структурные изменения
(добавить/удалить компонент, создать/уничтожить сущность) напрямую запрещены —
по умолчанию они бросают исключение. Почему: системы прямо сейчас идут по плотным
массивам компонентов, и если посреди обхода массив перетрясти, итератор сойдёт с ума.
Технически защита — счётчик глубины `_iterationDepth` в `World`.

**InitializeSystems** (`Systems/InitializeSystemRunner.cs`) — «инициализатор» новых
компонентов. Когда вы добавляете компонент, его `Awake`/`Start` вызываются не сразу,
а здесь, в начале следующего `Update` (и только если сущность активна). Это
аналогично тому, как в Unity `Start` вызывается не в момент `AddComponent`.

Фазы `FixedUpdate` и `LateUpdate` устроены так же, но с тремя **асимметриями**,
о которые легко споткнуться:

1. `BeforeAll` выполняется **только** в начале `Update`.
2. `AfterAll` выполняется **только** в конце `LateUpdate`.
3. `InitializeSystems` (то есть `Awake`/`Start` новых компонентов) выполняется
   **только** в `Update`. Мир, который крутят одним лишь `FixedUpdate`, никогда не
   инициализирует новые компоненты!

### Frame и Version — два разных счётчика

У мира два независимых счётчика, и их легко перепутать:

- `World.Frame` — номер кадра. Увеличивается на единицу при **каждом** вызове фазы
  (`Update`, `FixedUpdate`, `LateUpdate` — каждый из них «кадр»).
- `World.Version` — «водяной знак изменений». Увеличивается только когда компонент
  становится готовым (`MarkReady`) или когда его «толкнули» (`Bump`). По этому
  счётчику работают реактивные запросы `[New]` и `[Bumped]` (глава 9): система
  запоминает, до какой версии она уже всё видела, и в следующий раз смотрит только
  на то, что новее.

`World.CurrentPhase` хранит, какая фаза идёт сейчас, а `World.AfterCurrentPhase`
возвращает командный буфер «после текущей фазы» — удобно, когда код не знает, из
какой фазы его позвали, но хочет отложить операцию «на конец вот этого прохода».

## 5. Сущности

Файлы: `Entities/EosEntity.cs`, `Entities/EntitiesContainer.cs`,
`Extensions/EntityExtensions.cs`.

### Что такое EosEntity

```csharp
public readonly struct EosEntity
{
    public readonly int Id;        // номер
    public readonly ushort Version; // «поколение» этого номера
    internal readonly World World;  // в каком мире живёт
}
```

Это **структура** (тип-значение): копируется при передаче, не лежит в куче, не
создаёт мусора для сборщика. Сущность — это буквально номер плюс поколение плюс
ссылка на мир. Имя (`entity.Name`) не хранится в структуре — оно каждый раз
запрашивается у мира.

Зачем `Version`? Номера сущностей **переиспользуются**: когда сущность №5 умирает,
номер 5 возвращается в свободный пул и достанется кому-то следующему. Если бы
сущность была просто числом, старый сохранённый «указатель» на покойную №5 вдруг
начал бы указывать на нового жильца. Поэтому при каждом уничтожении поколение номера
увеличивается, и проверка живости выглядит так: «номер существует И поколение
совпадает» — это и есть `entity.IsValid`. Устаревшая ручка просто перестаёт быть
валидной, а не указывает на чужака.

`EosEntity.Null` — специальное «пустое» значение (`Id = -1`). Важная тонкость:
`default(EosEntity)` — это **не** `Null` (у default `Id = 0`), поэтому в коде ядра
проверка «нет сущности» — это всегда `World == null`, а не сравнение с `Null` по Id.

### Создание и уничтожение

```csharp
var entity = new EosEntity(world, "Player");  // создана НЕАКТИВНОЙ
entity.On();                                   // включить
// ...
entity.Destroy();
```

Две вещи, которые надо знать:

1. **Сущности создаются неактивными по умолчанию** (`active = false` в конструкторе).
   Пока сущность неактивна, её компоненты не получают `Awake`/`Start` и не
   участвуют в системах — они «ждут включения». Это позволяет собрать сущность
   целиком (накидать компоненты, настроить) и только потом включить одним `On()`,
   чтобы вся инициализация увидела уже готовую сущность. Исключение: сущности,
   созданные через командный буфер (`buffer.Create(...)`), сразу активны.
2. **Активность иерархична**, как в Unity: `IsActive` — *эффективное* состояние
   (свой флаг И флаги всех предков), `IsActiveSelf` — только свой флаг. Выключили
   родителя — всё поддерево молча «заснуло»; включили обратно — проснулось.
   В сохранение пишется именно `IsActiveSelf`, а эффективное состояние
   пересчитывается из связей.

Внутри сущности живут в `EntitiesContainer`: список живых, стек свободных номеров
для переиспользования, разрежённые массивы «существует ли номер» и «какое у номера
поколение». `world.Entities.All()` отдаёт перечислитель по живым сущностям без
выделения памяти.

### Стабильные ключи

Номер сущности живёт только в рамках сессии — после загрузки сейва номера будут
другие. Если нужна «постоянная ручка» (например, чтобы квестовая система находила
сущность «деревенский_кузнец» в любой сессии), есть стабильные ключи:

```csharp
world.Entities.SetStableKey(entity, "village_blacksmith");
if (world.Entities.TryFind("village_blacksmith", out var smith)) { ... }
```

Ключ переживает сохранение/загрузку.

## 6. Иерархия: родители и дети

Файлы: `Hierarchy/HierarchyContainer.cs`, `Hierarchy/ParentChanged.cs`,
`Extensions/HierarchyExtensions.cs`.

У сущностей есть нативная иерархия родитель-ребёнок — как у Transform в Unity,
только без всякой графики, чистые связи между сущностями. Пользовательский интерфейс —
методы-расширения из `EOS.Extensions`:

```csharp
child.SetParent(parent);          // привязать
child.Detach();                   // отвязать (то же: SetParent(EosEntity.Null))
var p = child.GetParent();
foreach (var c in parent.Children()) { ... }    // без выделения памяти
parent.GetChildren(list, recursive: true);      // собрать всё поддерево в список
bool b = child.IsDescendantOf(parent);
parent.DetachChildren();
var kid = parent.CreateChild("Kid");
```

Как это хранится (`HierarchyContainer`): разрежённые массивы по номеру сущности —
у каждого номера записан родитель, первый ребёнок, следующий/предыдущий «брат»
(внутрисписочные связи, так называемый intrusive linked list — список, звенья
которого встроены прямо в массивы, без отдельных объектов-узлов). Плюс два кэша:
`_branchActive` (эффективная активность ветки, чтобы `IsActive` был мгновенным)
и `_root` (корень дерева для каждой сущности — `GetRoot` это просто чтение из
массива, а не подъём по предкам). Единственная «дорогая» операция — перепривязка:
`SetParent`/`Detach` проталкивают новый корень и флаг активности вниз по всему
перемещённому поддереву.

Правила, которые стоит запомнить:

- **Перепривязка — не структурное изменение.** Её можно делать прямо из систем,
  посреди кадра, как и `SetActive` или теги. Каждое изменение связи рождает событие
  `ParentChanged { Child, OldParent, NewParent }` (про события — глава 13).
- **Уничтожение каскадно**: `entity.Destroy()` уничтожает всё поддерево, причём
  **дети раньше родителя** — компоненты ребёнка освобождаются, пока родитель ещё
  жив (это важно, например, для Unity-вью: дочерние объекты успевают вернуться в
  пул до того, как умрёт родительский GameObject). Хотите пощадить детей —
  сначала `DetachChildren()`.
- Циклы, привязка к самому себе, связи между разными мирами и протухшие ручки
  отклоняются с ошибкой в лог. Порядок детей не гарантируется.
- При сохранении связь хранится в `EntityRecord.ParentLocalId` и восстанавливается
  вторым проходом, когда все сущности уже созданы, — порядок записей не важен.

## 7. Компоненты

Файлы: `Objects/EosObject.cs`, `Objects/ObjectsContainer.cs`,
`Objects/Interfaces/IObjectUpdate.cs`.

### EosObject — базовый класс

Любой компонент — это класс, наследующий `EosObject`:

```csharp
public class Health : EosObject
{
    public int Value = 100;

    protected override void OnAwake()  { /* при инициализации */ }
    protected override void OnStart()  { /* после OnAwake */ }
    protected override void OnDispose() { /* при удалении */ }
}
```

Добавление и доступ — через методы-расширения сущности
(`Extensions/EntityExtensions.cs`):

```csharp
var hp = entity.Add<Health>();
var hp2 = entity.Get<Health>();
if (entity.TryGet<Health>(out var h)) { ... }
bool has = entity.Has<Health>();
entity.Remove<Health>();
```

Изнутри компонента те же операции доступны как защищённые методы (`Add<T>`,
`Get<T>`…) — они действуют на собственную сущность компонента.

### Жизненный цикл компонента

1. **Добавление.** `Storage<T>.Add(entity)` создаёт объект и регистрирует его
   в «пуле ожидающих» (`ObjectsContainer`).
2. **Инициализация.** В начале следующего `Update` `InitializeSystems.Run()`
   проходит по ожидающим: если сущность жива и активна — вызывает `Awake()`,
   потом `Start()`, потом `MarkReady` (объект «готов», срабатывает реактивный
   канал `[New]`). Если внутри `OnAwake`/`OnStart` вылетело исключение —
   ставится флаг `IsFailed`, и объект **никогда** не станет готовым (но игра
   не падает; ошибка в логе).
3. **Работа.** Готовый и включённый компонент виден системам и запросам. Если
   компоненту нужен собственный «апдейт» без системы, можно реализовать интерфейс
   `IObjectUpdate` / `IObjectFixedUpdate` / `IObjectLateUpdate` — тогда метод
   `OnUpdate(float deltaTime)` будет вызываться миром каждый кадр (см. шаг
   `Objects.Update` в главе 4).
4. **Удаление.** `Dispose()` — сначала освобождаются «прослеженные» ресурсы
   (см. ниже), потом `OnDispose`, потом объект снимается с учёта.

### Включённость

```csharp
public bool IsEnabled => IsAwaken && IsStarted && _enabled && Entity.IsActive;
```

Компонент «виден миру» (системам, запросам, пер-объектным апдейтам), только если он
проинициализирован, не выключен вручную (`SetEnabled(false)` / `Disable()`)
и его сущность эффективно активна. Любая из четырёх причин делает его невидимым.

### Trace — автоматическая уборка ресурсов

Если компонент владеет чем-то, что нужно освобождать (подписка, файл, что угодно
с `IDisposable`), не обязательно писать `OnDispose` — можно «проследить» ресурс:

```csharp
protected override void OnAwake()
{
    Trace(someSubscription);   // будет освобождено автоматически при Dispose
}
```

### Bump — «я изменился»

`Bump()` изнутри компонента (или `entity.Bump<Health>()` снаружи) — сигнал
«данные этого компонента поменялись». На него реагируют системы с параметром
`[Bumped] Health` (глава 9). Сигнал схлопывается до одного раза за кадр.

## 8. Хранилище компонентов

Файлы: `Storage/Storage.cs`, `Storage/ObjectsStorageMap.cs`,
`Objects/Interfaces/IPoolableObject.cs`.

### Sparse set — как компоненты лежат в памяти

Для каждого типа компонента существует своё хранилище `Storage<T>`. Это не словарь,
а структура данных «sparse set» (разрежённое множество). Объясняю на пальцах.

Есть два массива:

- **Плотный** (`_data`, `_owners`, …) — все живые компоненты типа `T` лежат подряд,
  без дырок. Рядом с каждым записано, какой сущности он принадлежит.
- **Разрежённый** (`_sparse`) — индексирован номером сущности: «у сущности №17 её
  компонент лежит в плотном массиве на месте 3».

Зачем так сложно? Получаем лучшее из двух миров:

- *Обход всех компонентов* — просто пройти плотный массив подряд. Идеально для
  систем (память читается последовательно, кэш процессора доволен).
- *Найти компонент конкретной сущности* — одно чтение из разрежённого массива.
- *Удаление* — приём «swap-remove»: на место удаляемого ставится последний элемент
  плотного массива, и массив снова без дырок (плюс правка `_sparse` для
  переехавшего). Порядок элементов при этом не сохраняется — поэтому нигде в EOS
  на порядок компонентов в хранилище полагаться нельзя.

Записи в разрежённом массиве проверяются по владельцу и поколению
(`_owners[slot] == id && _ownerVersions[slot] == version`) — протухшая сущность
не «найдёт» чужой компонент.

Здесь же живут реактивные отметки: `MarkReady` ставит компоненту версию добавления
(питает `[New]`), `Bump` — версию изменения (питает `[Bumped]`); обе берутся из
`World.NextVersion()`. У хранилища есть «максимумы» этих версий, чтобы реактивные
системы могли мгновенно понять «в этом хранилище для меня ничего нового» и не
сканировать его.

### Пул компонентов

Создание объекта в C# — это работа для сборщика мусора, когда объект умрёт. Если
компонент какого-то типа создаётся/удаляется часто (пули!), его можно сделать
переиспользуемым — пометить пустым интерфейсом-маркером:

```csharp
public class Bullet : EosObject, IPoolableObject { ... }
```

Тогда `Storage<Bullet>` заводит стопку отработанных экземпляров: `Remove` — как
обычно вызывает `Dispose` (ваш `OnDispose` отработает), затем сбрасывает служебные
флаги объекта и кладёт его в стопку; следующий `Add` достанет его оттуда вместо
`new`. Переиспользованный экземпляр проходит **полный** жизненный цикл заново —
`Awake`, `Start`, `MarkReady`, реактивный `[New]` сработает как для нового. Никаких
специальных хуков «меня взяли из пула» нет — пул прозрачен. Единственная
обязанность автора компонента: **чистить свои данные** — либо обнулять поля
в `OnDispose`, либо полностью инициализировать их в `OnAwake`, иначе новый «жилец»
получит мусор от прошлой жизни.

### ObjectsStorageMap — реестр хранилищ

`world.ObjectsStorages` — карта «тип → хранилище». `Get<T>()` лениво создаёт
хранилище при первом обращении и дополнительно индексирует его под каждым
интерфейсом, который тип реализует (это нужно для интерфейсных запросов в системах,
глава 9), и по сущностям-владельцам (для быстрого уничтожения).

Важное внутреннее правило (вдруг полезете туда руками): при `Reset()` карта хранилищ
**не очищается** — чистится содержимое каждого хранилища, но сами экземпляры
остаются. Причина: системы держат прямые ссылки на хранилища; подмени экземпляры —
и все существующие запросы молча отвяжутся от реальности.

## 9. Системы

Файлы: `Systems/EosSystem.cs`, `Systems/SystemsRunner.cs`,
`Systems/SystemsRunner.QueryBuilder.cs`, `Systems/SystemsRunner.TopologicalSort.cs`,
`Systems/Groups/`, `Attributes/`.

### Как объявить систему

Система — класс, наследующий `EosSystem`. **Регистрировать её нигде не нужно**:
при `World.Init()` ядро находит все неабстрактные подклассы `EosSystem` во всех
сборках и создаёт по экземпляру на мир.

Логика пишется в методе с именем `Execute`. И вот главный фокус EOS: **сигнатура
метода — это и есть запрос**. Какие параметры объявите — такие сущности система
и получит:

```csharp
class MoveSystem : EosSystem
{
    void Execute(Position pos, Velocity vel, float dt)
    {
        pos.Value += vel.Value * dt;
    }
}
```

Этот метод будет вызван по одному разу за кадр для **каждой** сущности, у которой
есть и `Position`, и `Velocity` (оба готовые и включённые). Писать циклы не надо —
обход делает ядро.

Полная таблица того, что можно ставить в параметры:

| Параметр | Что означает |
|---|---|
| `T : EosObject` | обязательный компонент: сущность подходит, только если он есть |
| `[Optional] T` | необязательный: придёт `null`, если компонента нет |
| `IFoo` (интерфейс) | «любой компонент, реализующий IFoo» |
| `[Each] IFoo` | то же, но если у сущности несколько подходящих — вызов на каждый |
| `[New] T` | реактивно: только в кадре, когда `T` стал готов (только что добавлен) |
| `[Bumped] T` | реактивно: только когда на `T` позвали `Bump()` |
| `EosEntity` | сама сущность-владелец |
| `float` | дельта времени |

Плюс фильтры-атрибуты на методе:

```csharp
[Include(typeof(Alive))]                  // только сущности, у которых есть Alive
[Exclude(typeof(Frozen))]                 // и нет Frozen
[WithTag("Enemy")]                        // теги: см. главу 15
void Execute(Health hp) { ... }
```

### Реактивные запросы — подробнее

`[New]` и `[Bumped]` — это способ написать «систему-реакцию» без событий: «когда
у сущности появилось здоровье — повесь полоску», «когда здоровье изменилось —
обнови цифру». Работает через версии (глава 4): система хранит курсор — «до какой
версии мира я всё видела», и каждый кадр смотрит только компоненты с версией новее.

Подводный камень: **курсор двигается, даже когда группа системы выключена**.
Если выключить группу на минуту и включить обратно, система не получит «пачку
накопившихся» срабатываний — всё, что произошло за время сна, для неё потеряно.
Это сделано осознанно (иначе включение группы вызывало бы лавину устаревших
реакций). Также новая система начинает с текущей версии — она не сработает на
компоненты, существовавшие до её появления.

### Фазы, включение, порядок

- Свойство `UpdateType` (переопределяемое) направляет всю систему в `Update`
  (по умолчанию), `FixedUpdate` или `LateUpdate`.
- `On()` / `Off()` включают и выключают систему (с колбэками
  `OnEnable`/`OnDisable`); `UpdateWhen()` — переопределяемое условие «работать ли
  в этом кадре».
- `Awake()` / `Start()` — одноразовые жизненные методы самой системы.

**Группы**: атрибут `[Group(typeof(MyGroup))]` относит систему к группе
(`SystemGroup` — пустой класс-маркер). Группы наследуются (вложенность через
наследование классов-маркеров) и включаются/выключаются деревом:
`world.SystemGroups.Disable<CombatGroup>()` выключит все системы группы и подгрупп.

**Порядок выполнения** внутри одного уровня — детерминированная топологическая
сортировка (упорядочивание «кто перед кем» по стрелкам зависимостей):

- `[UpdateAfter(typeof(Other))]` / `[UpdateBefore(typeof(Other))]` — явные стрелки;
- `[UpdateOrder(число)]` — приоритет при равенстве (есть константы
  `UpdateOrderPhase.BeforeAll` / `AfterAll` — «совсем первым» / «совсем последним»);
- дальше — имя типа, потом порядок обнаружения. Циклы зависимостей — исключение.

Итог: порядок всегда один и тот же от запуска к запуску, ничего не «плавает».

## 10. Два способа исполнения систем: рефлексия и кодогенерация

Файлы: `CodeGen/SystemRegistryGenerator.cs`, `CodeGen/GeneratedSystems.cs`,
`CodeGen/SystemShape.cs`, `CodeGen/SystemSignature.cs`.

Как ядро вообще вызывает ваш `Execute`, если сигнатуру вы придумали сами? Два пути,
дающие одинаковый результат:

**Рефлексия (по умолчанию).** При инициализации мир «рассматривает» каждый метод
через механизм рефлексии (возможность программы изучать собственные типы и методы
во время работы), понимает форму параметров и строит вызыватель. Работает из
коробки, ничего настраивать не надо. Минус: на каждый вызов собирается массив
аргументов `object[]`, и значения упаковываются (boxing) — это лишний мусор для
сборщика, на тысячах сущностей заметно.

**Кодогенерация (по желанию).** `SystemRegistryGenerator.Generate(...)` пишет
C#-файл, в котором для каждой системы сгенерирован обычный типизированный код вызова
без рефлексии и без выделений памяти. Сгенерированный реестр сам подключает себя
при загрузке сборки (через `[ModuleInitializer]` в `GeneratedSystems.Provider`),
и исполнитель систем начинает использовать его вместо рефлексии. В Unity-слое есть
меню, которое держит этот файл всегда свежим (см. Часть 2, глава про кодогенерацию).

Важно: если форма какого-то метода не поддалась генератору или реестр устарел,
этот конкретный метод тихо (с предупреждением в логе) откатывается на рефлексию —
ничего не ломается, просто медленнее.

Ограничение обоих путей: **открытые дженерик-системы не поддерживаются** (класс
`MySystem<T>` сам по себе нельзя обнаружить и создать). Нужна закрытая именованная
наследница: `class FooSystem : BarSystem<Baz>`.

## 11. Запросы снаружи: World.Query

Файлы: `Queries/WorldQueries.cs`, `Queries/EntityQuery.cs` (и `EntityQuery2/3`),
`Queries/QueryFilter.cs`, `Queries/QueryResult.cs`.

Системы — для логики «каждый кадр». А если по сущностям надо пройтись из кода вне
системного цикла — из UI, из MonoBehaviour, из редакторского окна? Для этого
`World.Query<…>()` — метод-расширение на `IReadOnlyWorld`:

```csharp
using EOS.Queries;

foreach (var health in world.Query<Health>())
    Debug.Log(health.Value);

foreach (var (pos, vel) in world.Query<Position, Velocity>())
    pos.Value += vel.Value;

int enemies = world.Query<Health>().WithTag("Enemy").Count();
if (world.Query<Boss>().TryFirst(out var boss)) { ... }
```

Поведение ровно как у системных запросов: видны только готовые и включённые
компоненты; при нескольких типах обход идёт по самому маленькому хранилищу (так
дешевле); сущность не повторяется. Фильтры — `.With<T>() / .Without<T>()` и
теговые `.WithTag / .WithoutTag / .WithAnyTag / .WithOneTag`. Завершители —
`Any()`, `Count()`, `First()` / `TryFirst(out)`, `ForEach(...)`, `ToList()`.

Сам перебор не выделяет память (запрос — структура со структурным
перечислителем). Реактивных `[New]`/`[Bumped]` здесь нет намеренно — это
системная механика с курсорами. И да: запрос только читает; структурное изменение
посреди перебора поймает та же защита, что и в системах.

## 12. Дженерик-компоненты

Компонент может быть дженериком — классом с параметром-типом. **Закрытый** дженерик
(с подставленным типом, например `Incarnation<GameObject>`) — это обычный компонент:
у каждого закрытого варианта своё хранилище.

Вопрос: как системе обойти *все* варианты сразу, не зная списка подставленных типов?
Идиома такая: дженерик-класс реализует обычный (недженериковый) интерфейс, а система
запрашивает интерфейс с атрибутом `[Each]`:

```csharp
public interface IIncarnation { void Sync(); ... }
public class Incarnation<TView> : EosObject, IIncarnation { ... }

class SyncSystem : EosSystem
{
    void Execute([Each] IIncarnation inc) => inc.Sync();
}
```

`[Each]` обязателен: интерфейсный запрос без него схлопывает по сущности (одно
срабатывание на сущность), а с ним — вызов на каждый подходящий компонент, даже если
у сущности их два разных закрытых варианта.

Сериализация закрытых дженериков работает без какой-либо регистрации: в снимок
пишется полное имя типа, а при загрузке оно разрешается терпимо к версиям сборок.

## 13. События

Файлы: `Events/EventChannel.cs`, `Events/EventsContainer.cs`.

События в EOS — это **одно-кадровые сообщения-структуры**: «произошёл урон»,
«сменился родитель». Идея: отправил сейчас — каждый заинтересованный прочитает ровно
один раз — событие исчезло. Никаких подписок, делегатов и утечек.

Отправка — из любого места, в том числе прямо из системы посреди кадра (это не
структурное изменение):

```csharp
public struct DamageEvent { public EosEntity Target; public int Amount; }

World.Event(new DamageEvent { Target = e, Amount = 5 });
```

Чтение — метод `EventExecute` в системе (можно несколько, на разные типы):

```csharp
class DamageSystem : EosSystem
{
    void EventExecute(DamageEvent e)  { /* по разу на каждое событие */ }
    void Execute(Health h)            { /* обычный запрос, идёт после событий */ }
}
```

Как устроен жизненный цикл (см. шаг `Events.Promote/Trim` в главе 4):

- Отправленное событие попадает в **буфер ожидания** и становится видимым только
  со **следующего** прохода фазы — события «всплывают через тик», момент чтения
  всегда одинаков и не зависит от того, какая система успела отправить раньше.
- «Ровно один раз» гарантирован персональным курсором каждого `EventExecute`
  (как у `[New]`): даже если `FixedUpdate` прокрутился три раза за кадр, повторного
  чтения не будет.
- Событие умирает, когда **все** зарегистрированные читатели его типа прошли мимо
  него курсором. Страховка от вечной жизни — жёсткий потолок возраста в 16 кадров.
- Система в выключенной группе курсор всё равно двигает — события для неё
  пропадают (та же логика, что у реактивных запросов, глава 9).

`EventExecute` всех систем выполняется **до** всех `Execute` в той же фазе
и сортируется тем же графом порядка (`[Group]`, `[UpdateBefore/After]`,
`[UpdateOrder]`).

## 14. Отложенные изменения: командные буферы

Файлы: `Systems/CommandBuffer/EntityCommandBuffer.cs`, `CommandChain.cs`,
`BoundSchedule.cs`, `DeferredEntity.cs`, `Core/StructuralChangePolicy.cs`.

Вспомним главу 4: пока работают системы, структурные изменения запрещены. А как же
системе, скажем, заспавнить пулю? Ответ: **отложить операцию в командный буфер** —
очередь, которая выполнится в безопасной точке кадра.

У мира восемь буферов — по точкам кадра: `BeforeAll`, `BeforeUpdate`, `AfterUpdate`,
`BeforeFixedUpdate`, `AfterFixedUpdate`, `BeforeLateUpdate`, `AfterLateUpdate`,
`AfterAll` (когда какой выполняется — см. схему кадра в главе 4 и три асимметрии).
Плюс `AfterCurrentPhase` — «после той фазы, что идёт сейчас».

Пишется это цепочкой (fluent-стиль — вызовы, нанизанные через точку):

```csharp
// из системы: создать пулю
var bullet = World.AfterUpdate.Create("Bullet")
    .Add<Position>(p => p.Value = muzzle)
    .Add<Velocity>(v => v.Value = dir * speed)
    .Entity;   // DeferredEntity — «обещание» сущности

// из системы: изменить чужую сущность с условием
World.AfterUpdate.Schedule(target)
    .When<Health>()             // только если есть Health…
    .WhenNoTag("Invulnerable")  // …и нет тега
    .Change<Health>(h => h.Value -= damage)
    .Destroy();                 // пример: и уничтожить
```

Семантика, которую нужно знать:

- Ложное условие (`When*`, `If`) **обрывает остаток своей цепочки** — последующие
  операции той же цепочки не выполнятся.
- `Add<T>` — «добавить, если нет» (колбэк настройки вызывается только при реальном
  добавлении). `Change<T>` — «изменить, только если есть». `ChangeOrAdd<T>` —
  всегда.
- `Create` возвращает `DeferredEntity` — «ещё-не-сущность», на которую уже можно
  навешивать дальнейшие операции; она превратится в настоящую (`.Value`), когда
  буфер выполнится. Созданная так сущность **сразу активна** (в отличие от прямого
  конструктора).
- Если операции, выполняясь, порождают новые отложенные операции — они выполнятся
  в том же проходе; от бесконечной цепочки защищает предохранитель на 10 000
  операций (буфер очищается, ошибка в лог).
- Цепочку можно собрать заранее как `CommandChain` и переиспользовать на разных
  сущностях через `Schedule(entity, chain)`.

Сама защита — `World.StructuralChangePolicy`: по умолчанию `Throw` (исключение),
можно переключить на `Warn` (предупреждение, но выполнить) или `Allow` (молча
разрешить). Менять политику стоит только осознанно.

## 15. Теги

Файлы: `Tags/TagsContainer.cs`, `Tags/TagRegistry.cs`, `Tags/TagFilter.cs`,
`Extensions/TagExtensions.cs`, `Attributes/TagAttributes.cs`.

Тег — лёгкая пометка на сущности без данных. Когда компонент с пустым телом —
перебор, тег — то, что нужно: «враг», «выделен», «в воде».

```csharp
entity.AddTag("Enemy");
entity.AddTag(UnitState.Stunned);     // ключом может быть и enum
entity.RemoveTag("Enemy");
bool b = entity.HasTag("Enemy");
bool any = entity.HasAnyTag("Burning", "Frozen");
```

Внутри это битовая маска на сущность: каждый ключ (строка или значение enum)
интернируется в номер бита реестром `TagRegistry`; `[Flags]`-enum раскладывается
по биту на каждый установленный флаг. Проверки — побитовые операции, очень дёшево.
Теги — не структурное изменение (можно из систем напрямую). В запросах систем —
атрибуты `[WithTag] / [WithoutTag] / [WithAnyTag] / [WithOneTag]`, в императивных
запросах — одноимённые методы. В сохранение теги пишутся по описанию (строка либо
тип enum + значение), так что переживают перезапуск.

## 16. Контекст и сервисы

Файлы: `Core/WorldContext.cs`, `Core/ServiceContainer.cs`, `Loader/WorldBootstrap.cs`.

Не все данные принадлежат какой-то сущности. Для «общемировых» данных два механизма.

**Контекст** (`world.Context`) — типизированная доска **структурных** значений:

```csharp
public struct GameSettings : ISerializableContext { public float Difficulty; }

world.Context.Set(new GameSettings { Difficulty = 1.5f });
var s = world.Context.Get<GameSettings>();
```

Внутри системы свойство `Context` даёт расширенную версию с вопросом «а изменилось
ли значение с тех пор, как я смотрела в прошлый раз?» — `Context.Changed<GameSettings>(out var s)`.
В сохранение попадают только значения, реализующие `ISerializableContext`.

**Сервисы** (`world.Services` / `world.ServiceRegistry`) — локатор сервисов на мир:
для объектов-одиночек с логикой (классов, не структур):

```csharp
world.ServiceRegistry.Register(new PathfindingService(...));
var nav = world.Services.Get<PathfindingService>();
```

Правила: регистрировать/снимать во время итерации нельзя (ошибка в лог) — сервисы
подключаются до запуска кадров; `Get` отсутствующего сервиса — ошибка в лог и null;
повторная регистрация того же типа молча заменяет старый сервис.

**WorldBootstrap.Provider** — шов «как засеять новый мир». Это статическая ссылка
на функцию `Action<World>` в `Loader/WorldBootstrap.cs`. Каждый `World.Init()`
**и каждый** `World.Reset()` прогоняют мир через неё. Ядро её никогда не заполняет —
это делает внешний слой (в Unity — сгенерированный код по атрибутам
`[EosWorldBootstrap]`, см. Часть 2). Смысл: дефолты контекста и сервисы
переустанавливаются автоматически для любого мира, в том числе после сброса.

## 17. Инкарнации: шов между логикой и картинкой

Файлы: `Objects/Incarnation.cs`, `Objects/IIncarnation.cs`,
`Loader/IIncarnationBinder.cs`, `Loader/IncarnationBridge.cs`,
`Systems/Incarnation/IncarnationSyncSystems.cs`, `Systems/Incarnation/IncarnationGroup.cs`.

«Инкарнация» (воплощение) — так в EOS называется визуальное представление сущности.
Ядро не знает, что такое GameObject, но умеет сказать: «у этой сущности должно быть
воплощение с идентификатором `"Units/Knight"`», — а внешний слой решает, что с этим
делать.

Действующие лица:

- **`Incarnation<TView>`** — компонент «у меня есть вью типа TView». `Setup(id)`
  запоминает строковый идентификатор. При `Awake` компонент находит **байндер**
  для типа `TView` и просит создать вью; при `Dispose` — уничтожить. Если байндера
  нет, инкарнация спокойно живёт без вью (отметка в лог на уровне Debug).
- **`IIncarnationBinder<TView>`** — интерфейс байндера («связывателя»): создать вью
  по идентификатору, уничтожить, синхронизировать каждый кадр. Реализуется во
  внешнем слое (в Unity: достать префаб, инстанцировать…).
- **`IncarnationBridge`** — статический реестр байндеров:
  `Register<TView>(binder)` / `Unregister` / `Resolve` / `Reset`.
- **`IncarnationSync*System`** — три системы ядра (по одной на фазу), которые
  каждый кадр обходят `[Each] IIncarnation` и зовут `Sync` / `SyncFixed` /
  `SyncLate` — так вью получает шанс подтянуться к данным. Это, кстати, живой
  пример идиомы из главы 12: дженерик-компонент за недженериковым интерфейсом.

`Incarnation<TView>` сериализуем: в снимок уходит идентификатор, при загрузке вью
пересоздаётся. Как всё это выглядит со стороны Unity — большая часть Части 2.

## 18. Сериализация: сохранение и загрузка

Файлы: `Serialization/WorldSerializer.cs`, `Serialization/WorldLoader.cs`,
`Serialization/IObjectSerializable.cs`, `Serialization/Snapshot/*.cs`.

Сохранение в EOS — двухступенчатое: ядро умеет превратить вселенную в **снимок**
(snapshot — набор простых объектов-записей в памяти), а уже куда писать этот снимок
(JSON? бинарный файл? облако?) — решает внешний слой.

**Захват.** `WorldSerializer.Capture()` обходит каждый сериализуемый мир и собирает
`UniverseSnapshot`: записи сущностей (имя, флаг `IsActiveSelf`, стабильный ключ,
теги, родитель `ParentLocalId`), значения контекста, и по каждому типу компонентов —
«мешок» записей. `WorldSerializer.Save()` — удобство: захватить и отдать в хук
`WorldLoader.OnSave`.

**Восстановление.** `WorldSerializer.Restore(snapshot)` работает **в два прохода**:
сначала создаются все сущности (старые номера переводятся в новые через таблицу),
и только потом добавляются компоненты и восстанавливаются их данные. Зачем два
прохода: компонент может хранить ссылку на другую сущность, и к моменту его
восстановления та сущность уже гарантированно существует — порядок записей не
имеет значения.

**Что сохраняет компонент.** Сам факт «на сущности был компонент типа X»
восстанавливается всегда. А вот данные — по желанию, через интерфейс:

```csharp
public class Health : EosObject, IObjectSerializable
{
    public int Value;

    public Type DataType => typeof(int);
    public object SerializeData() => Value;
    public void DeserializeData(object data, IDeserializeContext ctx)
        => Value = (int)data;
}
```

`IDeserializeContext` даёт `Resolve(localId)` — превратить сохранённый номер
сущности в живую сущность нового мира (для компонентов со ссылками на сущности),
и доступ к `World`.

Исключения из сохранения — на трёх уровнях: мир (`World.IsSerializable`), сущность
(параметр `isSerializable` конструктора), значение контекста (нет
`ISerializableContext` — не пишется).

Напоминание из главы 3: `Universe.Boot()` сам зовёт `WorldLoader.OnLoad` и
восстанавливает не-null снимок — загрузка при старте уже встроена.

## 19. Логи, профилировщик, отладочная отрисовка

Файлы: `Logging/EosLog.cs`, `Profiling/EosProfiler.cs`,
`Profiling/AggregatedProfilerBackend.cs`, `Diagnostics/WorldDebug.cs`.

**Логи.** `EosLog` — статический журнал с кольцевым буфером на 1024 записи
(новые затирают самые старые). Три уровня: `Debug` (вырезается из сборок без
символа DEBUG), `Warning`, `Error`. Куда писать — решает обработчик `EosLog.OnLog`
(по умолчанию — консоль; Unity-слой подвешивает сюда вывод в Unity-консоль).
`Dump()` отдаёт накопленное. Конвенция всего проекта: источником всегда передаётся
`nameof(ТекущийКласс)`.

**Профилировщик.** `EosProfiler` — фасад над сменным бэкендом с парой
`Begin(label)` / `End()`. По умолчанию выключен и стоит пустой бэкенд — ноль
накладных расходов. Ядро уже само обвешано замерами: каждая фаза мира и каждое
тело системы идут через `EosProfiler.Sample(label)` (см. `using (EosProfiler.Sample(...))`
в `Core/World.cs`) — скоуп закрывается даже при исключении внутри. В комплекте
`AggregatedProfilerBackend` — копилка «сколько раз и сколько времени на каждую
метку» с текстовым отчётом `Dump()`. Unity-слой ставит сюда свой бэкенд с
ProfilerMarker-ами.

**Отладочная отрисовка.** Ядро не умеет рисовать — оно умеет только *попросить
нарисовать*: у `EosObject` и `EosSystem` есть пустой виртуальный `OnDebugDraw()`.
Внешний слой в нужный момент (в Unity — из `OnDrawGizmos`) зовёт
`Universe.DebugDraw()`, и вызов расходится по всем мирам, объектам и системам,
каждый — под `try/catch`, чтобы одна кривая отрисовка не убила весь проход.

**Диагностика.** `WorldDebug` (`Diagnostics/WorldDebug.cs`) — текстовые дампы:
`DumpUniverse()`, `DumpWorld(world)`, `DumpEntity(entity)`. Их использует кнопка
«Copy dump» в Unity-инспекторе мира.

## 20. Статическое состояние и его сброс

Файл: `Loader/EosDomainReset.cs`.

В ядре есть статические «долгожители», которые переживают любой мир: обработчик
логов `EosLog.OnLog`, хуки сохранения `WorldLoader.OnSave/OnLoad`, реестр байндеров
`IncarnationBridge`. В обычном приложении это не проблема, но в Unity-редакторе
домен может **не** перезагружаться между нажатиями Play (режим быстрого входа в
Play Mode) — и тогда статика прошлой сессии протекла бы в новую: двойные подписки,
протухшие байндеры.

Для этого есть `EosDomainReset.Reset()`: одним вызовом гасит вселенную
(`Universe.Shutdown()`) и чистит все перечисленные статические швы. Unity-слой
зовёт его в самом начале каждой игровой сессии (см. Часть 2, глава про запуск).

Единственное намеренное исключение — `WorldBootstrap.Provider` не сбрасывается:
он указывает на сгенерированный статический метод без состояния и переустанавливается
каждый запуск сам.

## 21. Словарик терминов

- **ECS (Entity-Component-System)** — архитектура: сущности (номера) + компоненты
  (данные) + системы (логика, обходящая компоненты).
- **Сущность (entity)** — идентификатор «игрового объекта»: номер + поколение + мир.
- **Компонент** — кусок данных на сущности; в EOS — класс, наследник `EosObject`.
- **Система** — класс с методом `Execute`, который ядро вызывает для каждой
  подходящей сущности каждый кадр.
- **Структура (struct)** — тип-значение в C#: копируется при передаче, не создаёт
  объекта в куче, не нагружает сборщик мусора.
- **Сборщик мусора (GC)** — механизм .NET, освобождающий ненужные объекты; его
  паузы заметны в играх, поэтому EOS старается не создавать лишних объектов.
- **Boxing (упаковка)** — превращение значения-структуры в объект в куче (например,
  при передаче через `object`); создаёт мусор. Причина, по которой существует
  кодогенерация (глава 10).
- **Sparse set (разрежённое множество)** — пара массивов «плотный + разрежённый»:
  быстрый последовательный обход + мгновенный поиск по номеру (глава 8).
- **Swap-remove** — удаление из плотного массива переносом последнего элемента на
  место удалённого; быстро, но меняет порядок элементов.
- **Структурное изменение** — операция, меняющая состав данных: создать/уничтожить
  сущность, добавить/удалить компонент. Запрещена посреди обхода систем.
- **Командный буфер (ECB, EntityCommandBuffer)** — очередь отложенных структурных
  изменений, выполняемая в безопасной точке кадра.
- **Итерация / охраняемая зона** — участок кадра, где системы обходят компоненты
  и структурные изменения запрещены.
- **Реактивный запрос (`[New]` / `[Bumped]`)** — параметр системы, который
  срабатывает только на недавно добавленные / недавно «толкнутые» компоненты.
- **Курсор / водяной знак (watermark)** — запомненный номер версии «до сих пор я
  всё видел»; основа реактивных запросов и событий.
- **Рефлексия** — способность программы изучать собственные типы и методы во время
  работы; удобно, но медленно и с аллокациями.
- **Кодогенерация** — заранее написанный генератором обычный код, заменяющий
  рефлексию ради скорости.
- **Топологическая сортировка** — упорядочивание элементов по стрелкам «этот раньше
  того»; так сортируются системы.
- **Инкарнация** — визуальное воплощение сущности (в Unity — GameObject из префаба).
- **Байндер (binder)** — объект, знающий, как создать/уничтожить/синхронизировать
  вью конкретного типа по идентификатору.
- **Шов (seam)** — заранее оставленная точка подключения внешнего слоя к ядру.
- **Снимок (snapshot)** — состояние вселенной, превращённое в простые объекты-записи
  для сохранения.
- **Локатор сервисов** — контейнер «один объект на тип», у которого спрашивают
  нужный сервис (`world.Services.Get<T>()`).
- **Интернирование** — замена повторяющегося значения (строки-тега, имени вида
  модуля) на маленький номер, с таблицей соответствия.
- **Fluent-стиль** — построение операции цепочкой вызовов через точку:
  `a.Schedule(e).When<X>().Add<Y>()`.
- **Идемпотентность** — свойство операции: повторный вызов ничего не меняет
  (например, `EosLoop.Boot()` дважды безопасен).
